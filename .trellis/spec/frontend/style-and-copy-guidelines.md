# 样式与文案

> 目标：保持 JetLinks UI 的 Ant Design Vue 风格、信息密度和业务文案语气，不引入脱离项目的视觉体系或开发态文字。

## 样式事实源

优先级：

1. Ant Design Vue 组件默认结构和状态。
2. `jetlinks-web-core/src/style.css` 中的全局 token。
3. `jetlinks-web-core/vite.config.ts` Less `modifyVars` 和主题 token。
4. Tailwind `jet-*` token 或项目已有工具类。
5. 目标模块相邻 Less/CSS。

不确定时先查相邻页面和 `jetlinks-web-style` 的 token 规则，不临时写一套视觉语言。

## 间距、圆角、阴影、颜色

- 功能块之间、标题组与主体内容、子区域之间的距离按语义选择 token 或相邻组件默认节奏。
- 不为了视觉对齐临时写孤立 `10px`、`14px`、`18px`、`22px`。
- 卡片、面板、抽屉、表格优先保留 Ant Design Vue 和现有组件默认间距。
- 不新增重型阴影、装饰渐变、独立品牌色、陌生圆角体系。
- 状态强调优先 badge、pill、tag、弱底、描边；不用单侧彩色线条作为主要状态信号。

## 页面视觉规则

- 一屏一主视觉锚点：主 CTA、关键数据带、真实产品状态区三选一。
- 不默认堆“4 KPI + 搜索 + 大表格”。
- 工作流页主要内容首屏可见，不用解释型 banner 或装饰卡把核心内容推下去。
- 统计卡、图表、趋势、排行必须有真实数据源、刷新机制、决策用途和动作承接。
- 不做纯装饰图标、假数据卡片、占位图表。

## 自定义 CSS

写 Less/CSS 前先判断是否能用组件 props、slots、token 或已有类解决。

允许自定义：

- 业务布局 geometry。
- 复杂局部交互状态。
- 模块内独有但有业务意义的视觉组织。

不允许自定义：

- 已有组件能覆盖的基础按钮、表单、表格、弹窗样式。
- 全局污染选择器。
- 深层覆盖 Ant Design 内部结构且无注释说明的样式。
- 大量 magic number。

## 用户可见文案

写标题、按钮、Tooltip、空态、校验、弹窗描述前，先识别：

- 最终用户角色。
- 用户目标动作。
- 用户能理解的业务术语。
- 用户下一步应该做什么。

任何一项不清楚且会影响文案含义时，先问用户。

## i18n 规则

- 用户可见文案统一走当前模块 i18n。
- 模块文案放 `modules/<module>-ui/locales/lang/zh.json` 和 `en.json`。
- 主应用共享文案放 `jetlinks-web-core/src/locales/lang/*.json`。
- 字段 key、接口参数、路由参数不是展示文案，不需要翻译。
- 相邻模块无 i18n 时，先沿用其事实并说明风险，不自造新机制。

## 空态、错态、加载态

- 空态要说明对象和下一步动作，例如“暂未接入设备，先创建一个产品再添加设备”。
- 错态要说明失败对象、可能原因和可执行动作。
- 加载态尽量靠近被加载区域，不用全页阻断小范围刷新。
- 校验文案给具体限制和示例，不写“输入有误”这种空泛提示。
- 成功反馈尽量包含对象名和影响范围。

## 禁止文案

最终界面不得出现：

- `TODO`
- `待接接口`
- `占位文案`
- `待定`
- `这里以后接数据`
- `原型说明`
- `交互方式`
- `设计意图`
- `本功能用于实现...`
- 字段 key、变量名、API 路径、组件名、Topic 名、命令 ID、权限码、内部 schema 名
- 用户原始 prompt 或需求文档原文

线框图和设计备注只用于对齐，不进入最终界面。

## 反模式

- 为了“高级感”新增和当前 Ant Design 风格不一致的装饰背景。
- 顶部 banner、空态、Tooltip、帮助入口重复同一段说明。
- 把最终用户看不懂的技术词当说明文案。
- 用大段文字解释功能，而不是通过状态、控件、空态和就近提示引导。
