# 页面方案选择

> 目标：新增页面或调整主结构前，先判断业务第一任务和交互方案，避免默认套“传统后台表格 + 编辑弹窗”。

## 何时必须建立方案档案

除局部调整白名单外，以下任务必须先形成 solution profile：

- 新增页面。
- 重构页面壳层。
- 调整信息架构。
- 调整主筛选、主列表、主详情承载方式。
- 用户要求“参考某页面风格”“复用现在这种风格”“更适合运营使用”“不要普通表格页”。
- 页面可能落成标准管理页、详情页、工作台、分步流、概览页、时间线、树表、主从详情等多种壳层。

局部调整白名单：

- 仅改一个表单字段。
- 仅改一个筛选条件。
- 仅改一个弹窗、抽屉或单个区块内容。
- 仅样式、文案、组件 props 微调。

如果一个字段改动同时影响列表列、详情分区和编辑流程，就不再是局部调整。

## 业务五问

实现前必须能回答：

1. 目标用户是谁。
2. 用户进入页面后的第一任务是什么。
3. 页面成功标准是什么。
4. 用户操作的是单对象还是对象集合。
5. 如果页面有统计卡、图表、趋势或指标，数据来自哪里，帮助谁做什么决策。

命中规则：

- 任一关键事实未知且影响首屏或结构：先问用户。
- 两项及以上未知：暂停实现，先澄清。
- 文字对齐成本高：先给低保真 ASCII 线框或结构方案。

## 常见页面分型

| 第一任务 | 推荐方案方向 |
| --- | --- |
| 录入、查询、批量操作、状态管理 | 标准管理页，但仍先判断是否需要 `ConditionFilter` |
| 快速识别一批对象状态并进入处置 | 资产卡片台账页 / 筛选工作台 |
| 持续维护单个对象 | 对象详情工作区 |
| 左侧对象集合，右侧详情或编辑 | 主从详情工作区 |
| 配置开通新能力 | 分步流 / 配置向导 |
| 告警、工单、巡检、异常处置 | 处置工作台 |
| 看整体态势、健康度、分布 | 监控感知页 / 分析钻取页 |
| 查看事件演进、日志、回放 | 时间线 / 日志流页 |
| 选择模型、模板、资源、能力 | 资源 / 能力选择器 |

不要因为业务名叫“管理”就直接套表格；也不要为了“好看”添加无来源的 KPI 或图表。

## Solution Profile 字段

方案锁定后至少记录：

- `solutionName`
- `primaryFilterSurface`
- `contentSurface`
- `detailCarrier`
- `editMode`
- `coreComponents`
- `actionPlacement`
- `densityTarget`
- `sidebarMode`
- `confirmationMode`
- `rejectedAlternatives`

当业务事实清楚且只有一个可信推荐方案时，可以记录“默认采用推荐方案”并继续实现。关键事实缺失、结构风险高或两个以上方案同样合理时，先让用户选择。

## 反传统后台感约束

实现时至少遵守：

- 编辑触发梯度：`inline > sectional > drawer > modal > page`。
- 首屏一屏一主视觉锚点，不堆“4 KPI + 顶部搜索 + 大表格”。
- 状态先于字段，列表/卡片/摘要先呈现状态、标签、归属、主动作。
- 禁止用单侧彩色线条作为主要状态信号。
- 弹窗不是唯一编辑路径。
- 详情页头部摘要区中，名称/说明/标签/状态优先就地编辑或快捷动作。
- 用户可见文案只面向最终用户，不出现设计备注、开发说明、原型标注。

详细交互模板以 `jetlinks-web-style` 的 catalog 为源；实现前仍要回到当前 workspace 组件事实核验。

## 参考页面规则

参考优先级：

1. 当前模块相邻页面。
2. 当前模块相邻组件/API。
3. 同业务域兄弟模块。
4. 其他业务模块的相似页面。

只借鉴业务任务相似的结构、区块节奏、状态反馈和组件组合；不要复制完全不同业务的指标、流程、快捷操作、局部组件或样例数据。

## 禁止事项

- 跳过方案档案直接默认 CRUD。
- 把方案选择当作组件可用性证明。
- 从旧项目、skill 文档或参考页面发明当前 workspace 不存在的组件 props。
- 添加没有数据来源、刷新机制、决策用途和动作承接的统计卡、趋势图、排行榜。
- 把 wireframe、设计说明、交互原则渲染到最终页面。
